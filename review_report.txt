Determinism firewall analysis complete. I read the sim contract, `render.rs`, `net_loop.rs`, plus `lockstep.rs` and `sim::step`.

## Verdict: the firewall holds. Here is WHY.

**Structural guarantee (the strongest reason).** `Lockstep` exposes sim mutation through exactly three methods ‚Äî `submit_local_input`, `record_remote`, `try_advance` ‚Äî and a single read-only accessor `sim(&self) -> &Sim` (lockstep.rs:258). There is no `sim_mut`. The client physically *cannot* reach sim state any other way; the borrow checker enforces it.

- `drive_lockstep` is the **only** sim writer (render.rs:306‚Äì391). Everything else (`apply_transforms`, `spawn_world`, `update_hud`, `SimSnapshot::capture`) takes `&Sim`/`NonSend` and writes only Bevy `Transform`s / HUD text. ‚úì
- `state.prev` (`SimSnapshot`) is a render-only shadow copy, overwritten from the authority each tick, never read back into the sim (render.rs:321). ‚úì

**The f32 ‚Üí wire firewall (your "client value leaking into Input" concern).** Yes, the local input is computed in f32 ‚Äî `look_axis = pending.yaw_delta / max_yaw_per_tick_radians()` (render.rs:323), mouse delta, gamepad, `dt`, all f32. But that f32 produces **only the local player's own `Input`**, which is immediately quantized to integer i16 by `Input::new`/`q()` (sim.rs:138) and that i16 is the value `submit_local_input` records *and* broadcasts (lockstep.rs:145‚Äì157). No machine ever re-derives another machine's input from floats ‚Äî remote inputs arrive as wire i16 via `record_remote`. So even if two boxes compute a slightly different f32, the *quantized i16 each broadcasts* is the single authoritative value every peer applies. Float divergence at the boundary is invisible to the sim. ‚úì

**Yaw quantization matches the sim cap deterministically.** Client `max_yaw_per_tick_radians = (trig::TURN/24) as f32 / TURN * TAU` (render.rs:271) uses the *same integer* `trig::TURN/24` the sim's `MAX_YAW_TURNS_PER_TICK` does (sim.rs:210). Full deflection ‚Üí `look_axis=1.0` ‚Üí `q‚Üí1000=AXIS_SCALE` ‚Üí sim `dyaw = 1000 * (TURN/24)/1000 = TURN/24`. The sim's application (sim.rs:414‚Äì416) is pure integer. Test `full_look_axis_turns_one_tick_cap` pins it. The f32 cap constant is only a *normalization* of the local axis; the applied delta is integer. ‚úì

**ScriptedPeers cannot desync.** It feeds a constant integer `Input::new(0.0,1.0,0.0,0)` through `record_remote(pid, bot_msg)` at the *same* `apply_tick` as the local input (render.rs:347‚Äì352) ‚Äî byte-identical path to a wire peer; the sim can't distinguish source. Critically it's inserted **only in `build_screenshot_app`, which is always solo (`net=None`, single machine)** ‚Äî there is no second peer to disagree with, and no cross-peer hash comparison runs. The input is a compile-time constant, so even the solo run is reproducible. ‚úì (Caveat below.)

**Fixed-tick survives the variable-frame-rate driver.** `drive_lockstep` runs in `Update` but the accumulator subtracts *exactly* `TICK_DT` per step and calls `step` once per iteration (render.rs:315‚Äì316). The sim has no `dt` at all ‚Äî it's tick-counted ‚Äî so it never steps with a partial/wall-clock quantum. Frame rate only changes *how many* identical steps run per frame; `next_issue_tick` increments once per submit regardless of pacing, so tick *numbers* stay consistent across peers and `try_advance` stalls until all inputs land. ‚úì

## Minor issues ‚Äî NOT desyncs, but worth noting

- **Multi-tick catch-up drops look input on all-but-first tick.** `gather_input` runs once per frame; on a frame where the accumulator fires N ticks, the first tick drains `pending.yaw_delta` to 0 (render.rs:323‚Äì327) and ticks 2..N get `look_yaw=0` while reusing the same un-drained `strafe`/`forward`. Purely local feel (the broadcast i16 is still authoritative everywhere) ‚Äî no corruption, but the player's turn feels eaten after a stall.
- **`MAX_TICKS_PER_FRAME` backlog shed** (render.rs:382‚Äì385) drops *wall-clock* time, never partial ticks. A peer just falls behind real time and lockstep waits. No determinism impact. ‚úì
- **ScriptedPeers reuse hazard (latent).** If someone ever inserts `ScriptedPeers` into the *windowed* app (real peers present), every client would inject *its own* scripted input for the others, colliding with the real wire inputs at the same `apply_tick` ‚Äî `record_remote`'s `insert` would let last-writer-win per peer, and peers would disagree. Safe today only because it's screenshot-solo. A `debug_assert!(net.is_none())` guard would make that contract enforced rather than conventional.

No path mutates sim state outside the three sanctioned methods; no unbroadcast client float reaches the sim; the integer wire bytes are the shared truth.
